## Context

### 问题背景
BB_Math 项目在历史演进中尝试引入 MVP（Model-View-Presenter）架构模式，创建了 `MainFormPresenter` 和 `IMainFormView` 接口。但在实施过程中，由于某些功能问题（具体原因已不可考），开发团队回退到了原有的直接调用方式，导致：

1. **双重执行路径**：`button2_Click` → `answer()`（遗留模式）和 `button2_Click` → `Presenter.ValidateAnswer()` → `_view.RecordCorrectAnswer()`（MVP 模式）同时存在
2. **接口失效**：`IMainFormView` 接口中的 `RecordCorrectAnswer()` 和 `RecordWrongAnswer()` 方法被 Presenter 调用，但实际执行的是 `answer()` 方法中的直接 UI 操作
3. **维护困难**：修改答题逻辑需要在两个地方同步修改，容易遗漏
4. **文档脱节**：OpenSpec 规范可能基于 MVP 模式编写，与实际代码不一致

### 技术约束
- **架构模式**：项目已经决定采用 MVP 模式作为标准架构
- **向后兼容**：不能破坏用户现有的配置文件和历史记录
- **窗体保护**：不能修改 `*.Designer.cs` 文件
- **IDE 约束**：代码编写在 VS Code，编译调试在 VS 2019

### 相关历史记录
- `openspec/changes/archive/2026-01-13-fix-coins-and-timer-issues/session-summary.md` 提到："会话2：功能回归修复 - 恢复 answer() 方法调用"
- 这说明曾经尝试移除 `answer()` 方法但遇到了功能问题，又回退了

## Goals / Non-Goals

### Goals
1. **统一架构**：将所有答题功能迁移到 MVP 模式，消除双重路径
2. **清理代码**：标记或移除遗留的 `answer()` 方法及相关逻辑
3. **规范接口**：更新 `IMainFormView` 接口，移除未使用的方法
4. **文档同步**：确保 OpenSpec 规范与代码实现一致
5. **无回归**：确保所有现有功能在重构后仍能正常工作

### Non-Goals
1. **不修改 MVP 模式本身**：保持现有的 `MainFormPresenter` 和 `IMainFormView` 设计不变
2. **不改变业务逻辑**：答题奖励、惩罚机制等保持不变
3. **不新增功能**：本次重构只解决架构混乱问题，不添加新特性
4. **不优化性能**：性能优化不在本次重构范围内

## Decisions

### Decision 1: 保留而非删除 `answer()` 方法
**决策**：将 `answer()` 方法标记为 `[Obsolete]`，暂时保留代码，但不调用

**理由**：
- `answer()` 方法包含 286 行复杂逻辑，直接删除风险较高
- **重要**：`answer()` 方法经过充分测试，功能完全正常，重构时必须认真参照其逻辑
- 保留代码作为参考，便于回退和调试
- 如果重构遇到问题，可以快速切换回旧逻辑

**实施要求**：
- 在重构前逐行分析 `answer()` 方法的所有逻辑
- 确保 Presenter 路径完全复现 `answer()` 方法的功能行为
- 重点验证边界情况：全对通关、错误答题、超时、暂停、完成等

**替代方案考虑**：
- **直接删除**：风险过高，可能导致功能无法恢复
- **重构为单元测试**：工作量过大，超出本次重构范围
- **保留为注释**：不利于代码阅读，不如 `[Obsolete]` 标记

### Decision 2: Presenter 通过 IMainFormView 接口方法记录到 UI
**决策**：保留 `IMainFormView` 接口中的 `RecordCorrectAnswer()` 和 `RecordWrongAnswer()` 方法，Presenter 通过接口调用进行 UI 记录。

**理由**：
- WinForms 设计器生成的 TextBox 控件为 `private`，Presenter 无法直接访问
- 项目约束禁止修改 `*.Designer.cs` 文件，无法将控件改为 `internal` 或 `public`
- 接口方法提供了解耦的 UI 记录机制，符合 MVP 架构

**替代方案考虑**：
- **直接访问控件**：需要修改 Designer.cs，违反项目约束
- **事件机制**：过度设计

### Decision 3: 保留而非移除 IMainFormView 接口中的记录方法
**决策**：保留 `IMainFormView.RecordCorrectAnswer()` 和 `IMainFormView.RecordWrongAnswer()` 方法（参见 Decision 2）。仅保留已弃用的 Form1 接口实现在代码中供参考。

**理由**：
- Presenter 必须通过这些接口方法与 View 的私有控件交互
- 这些接口方法功能正确（在探索阶段已验证），只是之前未被调用路径触发

**替代方案考虑**：
- **移除接口方法**：Presenter 无法记录结果到 UI
- **标记为 `[Obsolete]`**：虽然可行，但会误导维护者

### Decision 4: 创建两个新的 OpenSpec 规范
**决策**：创建 `ui-architecture` 和 `answer-validation` 两个独立的规范

**理由**：
- `ui-architecture` 专注于架构模式，定义 MVP 的职责划分
- `answer-validation` 专注于业务流程，定义答案验证的详细步骤
- 单一职责原则，每个规范聚焦一个主题

**替代方案考虑**：
- **合并为一个规范**：会导致规范过于庞大，不利于维护
- **分散到现有规范**：现有规范（如 `scoring-system`、`time-management`）不够聚焦架构和流程

## Risks / Trade-offs

### Risk 1: 功能回归
**风险描述**：重构核心答题流程可能引入功能问题，导致答题功能失效或行为异常

**影响**：
- 用户无法正常练习数学题
- 金币奖励、惩罚机制可能出错
- 练习记录可能丢失或格式错误

**缓解措施**：
1. **全面测试**：按 `tasks.md` 中的测试清单逐一验证
2. **分步实施**：按照任务清单逐步推进，每步验证后再继续
3. **代码备份**：在重构前创建代码备份，确保可以快速回退
4. **保留旧代码**：`answer()` 方法标记为 `[Obsolete]`，紧急情况下可以临时恢复

### Risk 2: 遗漏功能点
**风险描述**：`answer()` 方法包含 286 行逻辑，可能遗漏某些边界情况或特殊处理

**影响**：
- 某些功能在新路径中失效（如全对奖励、暂停机制等）
- 定时器行为改变导致答题体验变差

**缓解措施**：
1. **逐行分析**：在任务 1.3 中详细分析 `answer()` 方法，列出所有功能点
   - ⚠️ **关键前提**：`answer()` 方法功能正常，必须确保完全理解其所有逻辑
   - 记录每个功能点的实现细节和边界条件
2. **对比检查**：将重构后的流程与原流程逐项对比
   - 创建详细的功能对比清单
   - 任何差异都需要调查和解释
3. **边界测试**：重点测试全对通关、错误答题、超时等边界情况
   - 验证行为与原 `answer()` 方法完全一致
   - 不允许任何功能差异

### Risk 3: 定时器逻辑错误
**风险描述**：定时器的启动、停止、重置逻辑在重构中可能出现问题

**影响**：
- 倒计时显示错误
- 超时惩罚机制失效
- 题型用时记录错误

**缓解措施**：
1. **保留定时器控制**：不修改定时器本身的实现，只改变调用时机
2. **验证定时器状态**：在测试中观察定时器的启动和停止
3. **日志验证**：检查日志中定时器相关的输出

### Trade-off: 代码冗余 vs 清晰性
**描述**：保留 `[Obsolete]` 的 `answer()` 方法会导致代码冗余

**权衡**：
- **优势**：降低重构风险，提供回退选项
- **劣势**：增加代码维护负担，可能误导维护者

**决策**：保留代码，但添加清晰的注释说明原因，建议在 3-6 个月后确认新路径稳定再删除
