# 重构提示词：统一 MVP 架构，消除双重逻辑混乱

## 任务背景

BB_Math 项目已开源发布，现在需要解决一个架构遗留问题。项目存在两条并行的执行路径：

1. **遗留模式（实际生效）**：`button2_Click → Form1.answer()`
2. **MVP 模式（未生效）**：`button2_Click → Presenter.ValidateAnswer() → _view.RecordCorrectAnswer()`

这导致：
- 维护困难（同一逻辑在两个位置实现但只在一个位置生效）
- 接口失效（Presenter 调用的接口方法未被执行）
- 文档脱节（OpenSpec 规范与实际代码不一致）
- 未来隐患（在 Presenter 中添加逻辑会被遗留路径绕过）

## 关键注意事项

### 高风险操作

1. **button2_Click 的修改**：这是答题的核心入口，修改错误会导致答题功能失效
2. **定时器处理**：定时器的启动、停止、重置逻辑必须正确
3. **全对奖励**：奖励逻辑涉及金币计算，必须确保准确
4. **存盘逻辑**：练习完成后必须正确保存，否则会丢失数据

### 回退策略

- 如果测试发现严重问题，立即回退到代码备份
- 保留 `answer()` 方法作为临时回退方案（标记为 Obsolete）
- 确保每个步骤都可以单独回退

### 兼容性考虑

- 不修改 `*.Designer.cs` 文件
- 不修改窗体控件和布局
- 不影响用户现有的配置文件
- 不影响历史练习记录
